# 프론트엔드 / UI 작업 규칙

사용자 대면 UI를 만들거나 수정할 때 지키는 규칙. 프로젝트 자체 컨벤션이 있으면 그쪽 우선.

## 1. 기존 자산 우선
- 프로젝트의 기존 UI 라이브러리·컴포넌트·디자인 토큰을 먼저 찾아 재사용.
- 이미 있는 컴포넌트를 복제하거나 새 컴포넌트를 남발하지 않음.
- 새 의존성 추가 전에 기존 스택으로 해결 가능한지 먼저 확인.

## 2. 필수 상태 커버리지
모든 데이터 기반 화면/컴포넌트는 4가지 상태를 항상 처리:
- **loading** — 로딩 중(스켈레톤/스피너)
- **empty** — 데이터 없음(안내 문구)
- **error** — 실패(원인·재시도 경로)
- **success** — 정상 데이터 표시

## 3. 반응형 기준
최소 3개 뷰포트(브라우저 표시 영역의 가로 폭)에서 레이아웃 확인:
- **375px** — 모바일
- **768px** — 태블릿
- **1440px** — 데스크톱

가로 스크롤 깨짐, 넘침, 겹침 없는지 확인.

## 4. 접근성
- 시맨틱 마크업 사용(`button`, `nav`, `main`, heading 계층 등).
- 키보드 포커스 이동·포커스 링 유지.
- 텍스트/배경 대비 확보.
- 이미지 `alt`, 인터랙티브 요소 `aria-*` 라벨.

## 5. 일관성
- 간격·타이포·색상은 디자인 시스템/토큰에서 파생.
- 매직 넘버·하드코딩 색상 지양.
- 기존 화면과 시각적 리듬(간격 스케일, 폰트 스케일) 통일.

## 6. 검증
- 변경 후 실제 렌더 확인(가능하면 Playwright MCP로 스크린샷 캡처).
- 브라우저 콘솔 에러/경고 0 목표.
- 위 상태(2)와 뷰포트(3)를 실제로 확인한 뒤 완료 보고.

## 7. 관련 라우팅
- UI/UX 문제 정리·개선안은 OMC `designer` 에이전트에게 요청.
